# ADR: Ultrawork Plan Document Quality Improvement

## Status

Accepted — 2026-03-20

## Context

Ultrawork plan documents (design documents generated by the planner agent) are critical inputs to the verification pipeline. Analysis of 14 plan documents across 4 projects (2025-11 to 2026-03) revealed:

- **Criteria specificity average**: 3.4/5.0 (7 of 14 documents contained subjective criteria like "normal behavior", "same functionality")
- **Context orientation average**: 3.7/5.0 (workers needed additional context exploration)
- **Cross-module impact documentation**: 0/7 projects tracked prop changes causing test regressions

Subjective criteria were the primary cause of verifier FAILs during the session, triggering repeated rework loops. The core problem: plan documents lack structure and rigor to guide deterministic verification.

## Decision

Implemented **Option B: SDD-Informed Redesign** — restructure 14 markdown files across ultrawork plugin along 4 axes to raise plan document quality from 4.3/5.0 to 4.7+.

### Design Decisions

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| D1 | Spec/Plan file separation | Logical separation (single document) | `tasks/*.json` already serves as execution unit; physical split unnecessary |
| D2 | Context Orientation position | First section of document | First information workers need before task decomposition |
| D3 | Subjective criteria handling | Banned + alternative patterns provided | Primary cause of verifier FAILs; requires explicit prohibition |
| D4 | Impact Analysis data source | Explorer collects during investigation | Planner collecting directly is token-inefficient |
| D5 | Self-containedness check location | Checklist in design template + planning SKILL.md guide | Enables planner self-verification and Codex doc-review re-verification |
| D6 | Data collection stage | Integrated into exploration (no separate stage) | Explorer already traverses codebase; separate stage adds overhead |
| D7 | Living document pattern | Not adopted | Documentation phase already exists; requires replan loop restructuring |
| D8 | Verification Strategy format | Criterion-Command-Expected Output table | Validated pattern from Vercel EDD + OpenAI ExecPlan specs |

## Architecture Notes

### Design Template Restructuring

The new design-template.md structure (from spec Appendix A) reorganizes plan documents for clarity:

```
# Design: {title}

## Overview
## Context Orientation        ← NEW: First context workers see
## Problem Statement
## Approach Selection
## Decisions
## Architecture
## Impact Analysis            ← NEW: Cross-module effects
## Scope
## Verification Strategy      ← NEW: Criterion-Command-Output
## Assumptions & Risks        ← MERGED: From separate headers
## Self-Containedness Checklist  ← NEW: Planner self-verification
```

Removed headers: Testing Strategy, Documentation (moved to post-verification phase)

### Criteria Standardization

Banned Korean expressions in success-criteria.md:
- ~~"기능 동일"~~ → "Command passes with exit code 0"
- ~~"정상 동작"~~ → "Output matches expected format"
- ~~"문제 없음"~~ → "Exit code 0, no error logs"

Pattern template:
```
## Criterion: {behavior}

Command: {exact bash command}
Expected Output: {deterministic output}
Exit Code: 0
```

### Verification Strategy Format

Criterion-Command-Expected Output table replaces narrative verification descriptions:

| # | Criterion | Command | Expected Output |
|---|-----------|---------|-----------------|
| V1 | Context Orientation section exists | `grep -c "## Context Orientation" file.md` | 1 |
| V2 | Impact Analysis section exists | `grep -c "## Impact Analysis" file.md` | 1 |

This format enables automated verification and tight coupling between design intent and executable tests.
